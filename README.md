# WH40k Homebrew Forge

A fully local browser workshop for creating, balancing, playtesting, and exporting unofficial homebrew units for Warhammer 40,000.

Homebrew Forge starts from an existing unit as a **proxy anchor**, records every change in a **Deviation Ledger**, estimates a transparent points range, and exports a clean two-page homebrew datasheet for opponent review.

No server, account, API key, installation process, or internet connection is required after downloading the files.

> **Unofficial fan project:** This project is not affiliated with, endorsed by, or sponsored by Games Workshop. Homebrew units are intended for friendly and narrative play with opponent approval.

## The three pieces

### 1. Homebrew Forge

The main application. It provides:

- A local project library with autosave
- Proxy-based unit creation
- Editable profiles, weapons, abilities, wargear, keywords, and points
- A proxy-relative balance estimate
- A Deviation Ledger showing every change from the anchor
- Damage and durability simulation against standard targets
- Playtest logs and version snapshots
- PDF, JSON, and portable `.hbf` exports
- A Data Pack Manager for faction packs, Tomes, and points overlays

### 2. Pack Foundry

A separate local conversion tool. It turns compatible BSData/New Recruit community catalogue files into `hbf-data-pack-2` packages understood by Homebrew Forge.

Pack Foundry can:

- Convert one faction catalogue
- Read an entire repository ZIP or extracted folder
- Resolve shared catalogue and library references
- Build separate faction packs
- Build a combined Tome
- Convert compatible MFM YAML files into points overlays
- Produce validation reports for unresolved or suspicious data

### 3. The Tome

The Tome is a single combined data pack containing many factions. It is convenient for browsing the full game from one installation.

The Tome is **reference data**, not a collection of editable homebrew projects. Installing it makes its units available in the Forge's proxy selector.

For maintenance, separate faction packs are usually easier to update. For convenience, the Tome is easier to install.

> Install either the Tome or the equivalent individual faction packs. Enabling both can produce duplicate proxy entries.

# Quick start

## Run Homebrew Forge

1. Download and unzip `Homebrew-Forge-V1.1.1.zip`.
2. Open `HomebrewForge.html` in a modern browser.
3. Open **Data Packs** from the sidebar.
4. Click **Install package**.
5. Select the Tome JSON or one or more faction pack JSON files.
6. Confirm that the installed pack is enabled.
7. Return to **Projects** and click **Create unit**.
8. Open **Forge**, choose a proxy, and click **Apply proxy baseline**.

Everything is stored locally in the current browser profile.

## Back up your work

Browser storage can be erased by clearing site data, using private browsing, changing browser profiles, or reinstalling the browser.

Use **Export `.hbf`** regularly. An `.hbf` file is the recommended portable project backup because it preserves the project and embedded image data.

# Proper Homebrew Forge workflow

The Forge works best when used as a documented design process rather than a blank datasheet editor.

## Step 1: Install reference data

Open **Data Packs** and install one of the following:

- The combined Tome
- One or more faction packs generated by Pack Foundry
- A manually authored `hbf-data-pack-2` file
- An optional `hbf-points-overlay-1` file

Installed faction packs appear under **Faction and core packs**. Points overlays appear separately.

Packages can be:

- Enabled or disabled
- Updated by importing a newer package with the same stable package ID
- Exported
- Removed

Disabling a pack hides it from the proxy selector without deleting it.

Removing or disabling a pack does not erase projects already created from it. The proxy baseline is copied into each project when the anchor is applied.

## Step 2: Create a project

From **Projects**, click **Create unit**.

Enter the high-level identity of the homebrew unit:

- Unit name
- Faction and optional subfaction
- Battlefield role
- Creator
- Project status
- Kitbash recipe
- Lore
- Optional unit image

The recommended project-status path is:

`Concept -> Untested -> Playtesting -> Revised -> Group Approved`

## Step 3: Apply the homebrew anchor

The **Proxy anchor** is the most important step in the workflow.

Choose:

1. The installed data pack
2. The faction
3. The closest proxy unit
4. The appropriate unit-size or points tier
5. The primary model profile

Then click **Apply proxy baseline**.

Applying the baseline:

- Copies the selected unit into the project
- Records package name, version, source ID, and status
- Stores the selected proxy points
- Creates the comparison snapshot used by the balance engine
- Starts the Deviation Ledger

### Choosing a good anchor

Choose the unit that is closest in actual battlefield function, not merely appearance or lore.

Good anchor criteria include:

- Similar model count
- Similar durability
- Similar movement
- Similar weapon role and range
- Similar support or utility role
- Similar deployment method
- Similar character, vehicle, monster, or infantry classification

A visually similar model with a completely different battlefield role is usually a poor anchor.

### Important anchor rules

- Apply the proxy **before** making substantial edits.
- Applying a new proxy can replace current profile, weapon, ability, and points fields.
- Reapply a proxy only when intentionally changing the comparison unit.
- **Adopt current unit as new baseline** resets the ledger around the current design. Use it only when starting a new revision phase, not to hide inconvenient deviations.
- Existing projects do not automatically change when a data pack or points overlay is updated. Reapply the updated proxy deliberately when you want a new baseline.

## Step 4: Make deliberate changes

Edit the imported unit rather than rebuilding it from zero.

Typical changes include:

- Renaming the unit or weapons
- Adjusting movement, toughness, save, wounds, Leadership, or Objective Control
- Adding or removing model profiles
- Changing attacks, skill, strength, AP, damage, or traits
- Adding structured or freeform abilities
- Changing unit size and proposed points
- Adding wargear, Leader, Transport, or attachment rules

Freeform abilities are permitted, but the Forge may not be able to score every interaction. Treat warnings as a request for human review rather than a prohibition.

## Step 5: Read the Balance Bench

The Balance Bench compares the current unit to the applied proxy.

It reports:

- Estimated power shift
- Suggested points range
- Offensive, defensive, mobility, utility, and board-control pressure
- Confidence level
- Combination warnings
- The full Deviation Ledger

Three balance profiles are available:

- **Friendly Narrative:** Wider tolerance for unusual or cinematic rules
- **Balanced Casual:** Recommended default for regular friendly games
- **Strict Matched Play:** Narrower tolerance and stronger warnings

The suggested range is a heuristic. It is not an official points value and should never override table evidence or opponent feedback.

## Step 6: Use the simulator

The simulator estimates average damage into common target profiles and supports several common offensive and defensive mechanics.

Use it to answer focused questions such as:

- Did the weapon gain too much output for its cost?
- Is the unit unusually efficient into both infantry and vehicles?
- Does an invulnerable save or Feel No Pain create a large durability jump?
- Is the unit strong only in its intended niche, or against nearly everything?

The simulator is a comparison tool, not a full battle engine. Mission play, positioning, target access, terrain, detachments, stratagems, and player decisions still matter.

## Step 7: Playtest and revise

Use **Playtests** to record:

- Opponent faction
- Mission
- Turns survived
- Damage dealt
- Units destroyed
- Your verdict
- Your opponent's verdict
- Notes and proposed changes

A useful minimum is several games against different army styles. Do not balance only from the unit's best or worst game.

Capture a **Snapshot** before major revisions. Snapshots preserve the design state and help explain why points or rules changed.

## Step 8: Export and share

The Export Workshop produces:

- **PDF:** A clean two-page datasheet and homebrew declaration
- **Print / Save as PDF:** Browser-native print workflow
- **`.hbf`:** Portable project backup and sharing format
- **JSON:** Human-readable project data

The PDF includes:

1. The playable homebrew datasheet
2. The proxy source, deviations, balance estimate, warnings, and opponent-agreement declaration

Share the second page with the first. The design history is part of the social contract.

# Data Pack Manager

## Installing a faction pack or Tome

1. Open **Data Packs**.
2. Click **Install package**.
3. Select the `.json` pack.
4. Confirm that it appears under **Faction and core packs**.
5. Enable the pack.
6. Return to **Forge** and refresh the proxy selection by choosing the pack and faction.

Importing a package with the same stable `id` updates the installed copy.

## Installing a points overlay

1. Install the underlying faction pack or Tome first.
2. Click **Install package** again.
3. Select the overlay JSON.
4. Enable it under **Points overlays**.
5. Check the compatibility report for unmatched package IDs or units.

Overlays replace points tiers and optional wargear costs without rebuilding unit profiles.

When several enabled overlays target the same pack, later matching values replace earlier values. In normal use, keep only the newest relevant overlay enabled.

## Updating reference data

Updating a package changes future proxy selections. It does not silently rewrite existing homebrew projects.

This preserves reproducibility: a project remains anchored to the profile and package version used when it was created.

To move an existing design onto newer reference data:

1. Create a snapshot or export an `.hbf` backup.
2. Install the updated pack or overlay.
3. Reopen the project.
4. Select the updated proxy and points tier.
5. Click **Apply proxy baseline**.
6. Reapply the intended homebrew changes and review the new ledger.

# Using Pack Foundry

## Run Pack Foundry

1. Download and unzip `Homebrew-Forge-Pack-Foundry-v0.2.zip`.
2. Open `PackFoundry.html`.
3. No server or internet connection is required.

## Recommended full-repository workflow

1. Download a compatible BSData repository using GitHub's **Code -> Download ZIP**.
2. Drop the untouched repository ZIP into Pack Foundry.
3. Choose **Full Tome** mode or select individual detected factions.
4. Review the detected catalogue checklist.
5. Keep **Safe linked-profile resolution** enabled.
6. Keep **Aggressive linked-group resolution** disabled unless investigating a known missing reference.
7. Forge the selected output.
8. Download the generated Tome or bundle ZIP.
9. Review the validation reports.
10. Install the generated pack in Homebrew Forge.

If repository ZIP import is unavailable in the browser, extract the repository and use **Choose Folder** instead.

## Single-faction workflow

For a smaller pack:

1. Load the faction catalogue JSON.
2. Load the shared game-system JSON and any faction library catalogues it references.
3. Select the faction catalogue as primary.
4. Forge the faction pack.
5. Review the validation report.
6. Install the output JSON in Homebrew Forge.

Missing library files often appear as unresolved links in the validation report.

## Points data

Compatible MFM YAML can be loaded separately, through an extracted folder, or from a ZIP containing YAML files.

Pack Foundry converts matching data into detachable `hbf-points-overlay-1` files.

Keeping points separate from profiles allows balance updates to remain small and replaceable.

## Validation reports

Generated packs should remain marked **Provisional** until reviewed.

Review warnings for:

- Missing points
- Missing unit profiles
- Missing weapons or abilities
- Unresolved entry links
- Suspiciously large numbers of linked profiles
- Incorrect model-count tiers
- Duplicate IDs or names
- Legends or legacy units
- Attachment relationships that could not be parsed

A successful conversion means the JSON is structurally usable. It does not guarantee that every source rule was interpreted perfectly.

# Creating data packs manually

Homebrew Forge supports two package schemas:

- `hbf-data-pack-2` for profiles, weapons, abilities, and relationships
- `hbf-points-overlay-1` for points and wargear-cost updates

The Forge still accepts older schema-v1 packs, but new packs should use schema v2.

## Minimal faction pack

```json
{
  "schema": "hbf-data-pack-2",
  "id": "my-faction-11e",
  "name": "My Faction Reference Pack",
  "version": "1.0.0",
  "edition": "11th",
  "status": "Provisional",
  "dependencies": [],
  "source": {
    "type": "Manual community pack",
    "revision": "1.0.0"
  },
  "shared": {
    "weapons": {
      "example-rifle": {
        "id": "example-rifle",
        "name": "Example rifle",
        "type": "Ranged",
        "range": "24",
        "a": "2",
        "skill": "3+",
        "s": 5,
        "ap": -1,
        "d": "1",
        "traits": ""
      }
    },
    "abilities": {}
  },
  "factions": [
    {
      "id": "my-faction",
      "name": "My Faction",
      "subfaction": "",
      "units": [
        {
          "id": "my-faction:example-squad",
          "name": "Example Squad",
          "role": "Battleline / Infantry",
          "status": "Provisional",
          "legends": false,
          "keywords": ["Infantry", "Battleline", "My Faction"],
          "modelProfiles": [
            {
              "id": "example-model",
              "name": "Example model",
              "primary": true,
              "baseSize": "32 mm",
              "stats": {
                "m": "6",
                "t": 4,
                "sv": "3+",
                "w": 2,
                "ld": "7+",
                "oc": 2,
                "inv": "",
                "fnp": ""
              }
            }
          ],
          "pointsOptions": [
            {
              "id": "models-5",
              "label": "5 models",
              "models": 5,
              "points": 90
            }
          ],
          "weapons": ["example-rifle"],
          "abilities": [],
          "wargearOptions": [],
          "wargearCosts": {},
          "leader": [],
          "attachmentTargets": [],
          "transport": [],
          "importWarnings": [],
          "source": {
            "id": "manual-example-squad",
            "catalogue": "Manual",
            "revision": "1.0.0"
          }
        }
      ]
    }
  ]
}
```

Weapons and abilities may be embedded directly in a unit, but shared references reduce duplication in large packs.

## Minimal points overlay

```json
{
  "schema": "hbf-points-overlay-1",
  "id": "my-faction-points-2026-07",
  "name": "My Faction Points Update",
  "version": "2026.07",
  "edition": "11th",
  "status": "Human Reviewed",
  "appliesTo": ["my-faction-11e"],
  "source": {
    "effectiveDate": "2026-07-15"
  },
  "entries": {
    "my-faction:example-squad": {
      "pointsOptions": [
        {
          "id": "models-5",
          "label": "5 models",
          "models": 5,
          "points": 95
        }
      ],
      "wargearCosts": {}
    }
  }
}
```

## Stable IDs

IDs are the stitching that keeps updates and overlays attached to the correct records.

Use stable IDs for:

- Packs
- Factions
- Units
- Model profiles
- Weapons
- Abilities
- Points tiers

Do not change an ID merely because a display name changed. Changing IDs causes overlays and external references to stop matching.

## Versioning and review status

Recommended package statuses:

- `Generated`
- `Machine Checked`
- `Human Reviewed`
- `Play Verified`
- `Provisional`
- `Outdated`

Use a version string that clearly identifies the data release, such as:

- `1.0.0` for a manually maintained pack
- `2026.07.2` for a dated community-data build

Preserve source provenance and conversion notes. The Forge displays this information in package reports and exported homebrew declarations.

# Preparing for future editions

The `hbf-data-pack-2` format was designed to be reusable, but the current Homebrew Forge build is specifically labelled and tuned for **11th Edition**.

A pack for a future edition can begin with the same schema when the game still uses compatible concepts such as:

- Movement
- Toughness
- Save
- Wounds
- Leadership
- Objective Control
- Ranged and melee weapon profiles
- Unit-size points tiers

To prepare a future-edition pack:

1. Copy an existing schema-v2 pack.
2. Give it a new stable package ID.
3. Change the `edition` field.
4. Update all unit, weapon, ability, and points data.
5. Preserve stable IDs across updates within that edition.
6. Install it and inspect the compatibility report.

The current Forge warns when a package identifies itself as an edition other than 11th. Its balance coefficients, interface labels, simulator assumptions, and PDF wording may also require an application update for a new edition.

If a future edition changes the characteristic model or weapon structure substantially, update Homebrew Forge and Pack Foundry before treating the new pack as fully supported.

In other words: the package format is a sturdy cargo crate, but a future rules engine may still require a new loading dock.

# Project files and storage

## `.hbf`

Portable Homebrew Forge project bundle. Recommended for backups and sharing projects between browsers or computers.

## Project JSON

Human-readable export of one project. Useful for inspection, version control, or external tooling.

## Faction pack JSON

Reference database installed through the Data Pack Manager. It does not create projects by itself.

## Tome JSON

A faction pack containing many factions in one file.

## Points overlay JSON

A detachable update that replaces points tiers and wargear costs for matching units.

## Validation report JSON

Pack Foundry's conversion notes. Keep reports beside generated packs so future maintainers can see unresolved issues.

# Troubleshooting

## The pack installed, but no proxy units appear

- Confirm the pack is enabled.
- Confirm that a faction is selected.
- Check the compatibility report.
- Make sure the file is a faction pack, not only a points overlay.

## Every unit appears twice

The Tome and equivalent individual faction packs are probably enabled together. Disable one set.

## Points look old or incorrect

- Confirm the intended points overlay is enabled.
- Disable older competing overlays.
- Check that the overlay's `appliesTo` value matches the pack ID.
- Remember that existing projects retain their previously applied baseline until deliberately updated.

## A project disappeared

Projects are stored in the current browser profile. Clearing browser data or using private mode can remove them. Restore the latest `.hbf` export.

## Pack Foundry attaches unrelated weapons to a unit

- Keep safe linked-profile resolution enabled.
- Keep aggressive linked-group resolution disabled.
- Review the unit's validation warnings.
- Convert the faction again with all required library catalogues loaded.

## Repository ZIP import fails

Extract the repository ZIP and use **Choose Folder** in Pack Foundry.

## A Tome is slow to install or browse

Use separate faction packs and enable only the factions needed for the current project.

## The points estimate seems wrong

Check the anchor first. A poor proxy choice can make a mathematically tidy comparison strategically meaningless. Then inspect unanalysed freeform abilities, attachment synergies, and playtest evidence.

# Design principles

Homebrew Forge follows a few deliberate rules:

1. **Local first:** User projects and imported data remain in the browser.
2. **Proxy relative:** Balance begins from an established battlefield role.
3. **Explain the estimate:** Changes are visible in the Deviation Ledger.
4. **Warn, do not forbid:** Strange ideas remain possible, but risks are identified.
5. **Playtests outrank arithmetic:** Table evidence is the final balancing instrument.
6. **Share provenance:** Opponents should know what the unit is based on and how it changed.

# Data sources and responsibility

Pack Foundry is designed to work with compatible community-maintained catalogue data, including BSData/New Recruit-style JSON and compatible MFM YAML.

Generated packs should retain:

- Source repository or dataset name
- Source revision
- Generation date
- Converter version
- Review status
- Validation warnings

Always review source licenses before redistributing converted data. Do not bundle official artwork, logos, or other protected assets without permission.

The Homebrew Forge software is released under the repository's [GPL-3.0 license](./LICENSE). Generated data packs may include data derived from separately maintained community sources and can be subject to their own terms.

# Disclaimer

Homebrew Forge does not certify tournament legality, official accuracy, or mathematical balance. All estimates are heuristic and proxy-relative.

Before using a homebrew unit:

1. Share the complete exported datasheet and declaration.
2. Agree on the proposed points with the opponent or group.
3. Record playtest results honestly.
4. Revise rules or points when the unit creates poor games.

The goal is not to win the spreadsheet. The goal is to make something memorable that both players are glad reached the table.
